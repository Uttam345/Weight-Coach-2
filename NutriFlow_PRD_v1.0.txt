1. Executive Summary
NutriFlow is a comprehensive personal wellness platform that unifies four distinct domains — nutrition tracking, health monitoring, gym workout management, and AI-assisted cooking — into a single, cohesive experience. The platform's defining USP is its AI Kitchen Cook module, which lets users track their kitchen inventory in real time and receive personalised meal suggestions, full recipes, and linked cooking videos, all generated or curated by AI.

This document covers the complete requirements lifecycle: high-level feature descriptions, detailed user stories, acceptance criteria, prioritisation, and concrete implementation guidance tuned for the Antigravity code-generation workflow with human review gates at each milestone.

Workflow Convention — Antigravity generates UI components, API integrations, and data models based on structured prompts. After each generation cycle the human reviewer (you) validates functional correctness, UX quality, and spec completeness before merging or iterating. Each implementation table in this document includes a dedicated Antigravity Prompt Hint column to guide generation, and a Human Review Focus column to guide validation.

2. Platform Overview
2.1 Vision Statement
To be the most intelligent personal wellness companion — one that understands what you eat, how you move, and what you have in your kitchen, and uses that understanding to help you live better every day.
2.2 Core Modules
	•	Module 1 – Nutrition Tracker: Log daily meals, macronutrients, calories, water intake, and dietary goals with a rich food database.
	•	Module 2 – Health Dashboard: Track vitals (weight, BMI, blood pressure, sleep, steps) and visualise trends over time.
	•	Module 3 – Gym & Workout Flows: Plan, log, and review workouts with guided exercise flows, progressive overload tracking, and rest timers.
	•	Module 4 – AI Chat Assistant: An always-available conversational AI to answer nutrition questions, workout advice, and general wellness queries.
	•	Module 5 – AI Kitchen Cook (USP): Track kitchen inventory, receive AI-generated meal options using available ingredients, get full recipes with macros, and watch linked cooking videos.

2.3 Target Users
	•	Fitness enthusiasts who want integrated tracking across nutrition and workouts.
	•	Home cooks looking to reduce food waste by cooking with what they already have.
	•	Health-conscious individuals who want personalised, data-driven wellness insights.
	•	Beginners who need guided flows and AI coaching to build healthy habits.

7.2 AI Meal Suggestion — Data Flow
Understanding the data flow is critical for Antigravity to generate correct integrations. The process runs as follows: the client sends the user's current inventory list (ingredient names + quantities), daily calorie remaining, dietary restrictions, and cuisine preference to the backend. The backend constructs a structured prompt for the Claude API requesting a JSON array of meal suggestions. Each suggestion object contains the meal name, ingredient list with required quantities, step-by-step instructions, estimated macros, and a search query string for the cooking video lookup. The backend then calls the YouTube Data API v3 using the search query and appends the top video result (title, thumbnail, videoId) to each suggestion before returning the enriched payload to the client.

Antigravity Prompt Hint for the Meal Engine: 'Create a FastAPI endpoint POST /api/kitchen/suggest. It accepts a KitchenSuggestRequest body with fields: inventory (list of {name, quantity, unit}), caloriesRemaining (int), dietaryRestrictions (list[str]), cuisinePreference (str). Build a buildMealPrompt() function that serialises this into a Claude API call requesting a JSON array of MealSuggestion objects. Each MealSuggestion has: title, ingredientMatch (float), ingredients (list of {name, qty, unit}), missingIngredients (list), steps (list[str]), macros ({calories, protein, carbs, fat}), videoSearchQuery (str).'

10. Antigravity + Human Review Workflow
10.1 Development Cycle
Each feature follows a four-stage cycle designed to maximise what Antigravity generates correctly on the first pass while keeping the human reviewer's effort focused and efficient.

	•	Stage 1 – Prompt Construction: Use the Antigravity Prompt Hints from this document as the base. Append specific constraints: component name, props interface, expected API shape, and any edge cases to handle. The more precise the prompt, the fewer review iterations needed.
	•	Stage 2 – Generation & Initial Review: Antigravity generates the component or endpoint. Run it locally immediately. Check the Human Review Focus column for that feature — these are the first things to validate.
	•	Stage 3 – Functional Verification: Execute the acceptance criteria from the feature spec as manual or automated tests. If a criterion fails, refine the prompt and regenerate only the failing part, not the entire file.
	•	Stage 4 – UX Review: As the human reviewer, evaluate the generated UI for: visual hierarchy, tap target size, loading state quality, error state handling, and dark mode correctness. Log issues as annotated screenshots and feed them back into the next Antigravity prompt.

10.2 Prompt Writing Best Practices for Antigravity
	•	Always specify the exact component name, file path, and framework (React Native / Next.js / FastAPI).
	•	Include the TypeScript interface or Pydantic model for all props / request bodies — Antigravity outputs cleaner code when types are explicit.
	•	Mention error states, loading states, and empty states explicitly in the prompt — they are commonly omitted without instruction.
	•	For AI integration prompts, always specify: model name, max_tokens, whether to stream, and the expected JSON schema of the response.
	•	After each review cycle, prepend a brief changelog to the prompt: 'Previous version had X issue. Fix by doing Y.' This anchors Antigravity's regeneration correctly.

	•	OQ-1 – Monetisation: Will the app be free, freemium, or subscription-based at launch? This affects how AI API costs are managed and whether usage limits apply to the Kitchen Cook module.
	•	OQ-2 – Indian Food Database: Open Food Facts has partial coverage for regional Indian dishes. Decision needed: build a custom seed dataset for common Indian foods and recipes, or integrate a paid API (Nutritionix has Indian food support).
	•	OQ-3 – Video Licensing: YouTube links are acceptable for MVP. Post-MVP, evaluate whether to build a curated in-app video library to avoid API quota limits and improve content quality control.
	•	OQ-4 – Wearable Support: Garmin and Fitbit integrations are not in scope for MVP. Decision needed on roadmap priority relative to platform maturity.
	•	OQ-5 – Recipe Content Generation vs Curation: The current design uses AI to generate recipes on-the-fly. An alternative is to curate a recipe database and use AI only for matching. Hybrid recommended: curated DB as primary, AI generation as fallback.
